# SDR 0003: Модель даних, маршрутизація та управління станом

## Контекст
Потрібно визначити, як застосунок перемикатиметься між різними екранами, як зберігатиметься структура окремої речі та як управляти станом "Активна позика / Повернуто" без видалення інформації.

## Рішення
1. **Маршрутизація (Routing):** Вирішено створити застосунок типу Single Page Application (SPA) без складної маршрутизації. Уся взаємодія (додавання, пошук, фільтрація, перегляд списку) відбувається на одній HTML-сторінці (index.html).
2. **Модель даних:**
   Записи зберігаються у вигляді масиву об'єктів. Схема об'єкта `Item`:
   - `name` (String): Назва речі.
   - `category` (String): Тип речі (Техніка, Книги тощо).
   - `friend` (String): Ім'я позичальника.
   - `lendDate` (String/Date): Дата видачі.
   - `returnDate` (String/Date): Очікувана дата повернення.
   - `returned` (Boolean): Статус архівування (false = активна, true = повернуто).
   - `actualReturnDate` (String/Date | null): Фактична дата повернення.

3. **Управління станом:** Замість видалення об'єкта (CRUD - Delete), реалізовано зміну статусу (Soft Delete). При натисканні кнопки "Повернути", властивість `returned` змінюється на `true`, що візуально переводить картку в сірий стан і переміщує її в кінець списку.

## Наслідки
- **Позитивні:** Зберігається повна історія всіх позик для користувача. Спрощується логіка візуалізації на одному екрані.
- **Негативні:** Відсутність окремих сторінок (URL) для кожної речі не дозволяє поділитися посиланням на конкретну картку.
